Architecture Overview
Stack
Next.js 14 App Router with server-side rendering and API routes. TypeScript end-to-end. Client components use React 18 with Suspense boundaries and skeleton loaders. Realtime updates via Supabase Postgres Change subscriptions.
Next.js 14React 18TypeScript 5App Router
Database & Auth
Supabase (PostgreSQL) with Row Level Security on all 7 tables. Supabase Auth handles JWT sessions via httpOnly cookie (pharmatrack_token). Auth emails (signup confirmation, password reset) route through Resend SMTP.
Supabase PostgreSQLRLSResend SMTP
Rate Limiting
Next.js middleware intercepts four API paths. Upstash Redis (sliding window, distributed across Vercel instances). Fail-open: if Redis is unreachable, falls back to in-memory Map so scan traffic is never blocked by a Redis outage.
Upstash RedisSliding WindowFail-open
Monitoring & Performance
Sentry integrated (client + server + edge configs). Production-only, no PII, 10% transaction sampling. 7 PostgreSQL indexes applied on hot columns (attendance_records, events, users).
Sentry7 DB IndexesRealtime Subscriptions
Email Infrastructure
Resend SDK handles all transactional emails. Event broadcasts go out in batches of 100 (8 API calls for 800 students). Fire-and-forget: API returns after DB insert, emails go out async. Auth emails (confirm, reset) route through Supabase's custom SMTP using Resend — removing the 3/hr free-tier cap.
Resend Batch APICustom SMTPFire-and-forget

Auth Callback /auth/callback
Handles two cases. Case 1 – Email OTP: token_hash + type → verifyOtp() → redirect to /login?verified=true (or /reset-password for recovery links). Case 2 – PKCE / OAuth: code → exchangeCodeForSession() → checks if profile exists → routes to /onboarding (new) or /dashboard (returning). Sets pharmatrack_token httpOnly cookie on success.

Authentication Flow — End to End
1
User submits registration form
Browser POSTs to POST /api/auth/register with email, password, full_name, role, and student fields. No browser-side Supabase call.
src/lib/auth-client.ts → registerStudent() / registerFacilitator()
2
Server calls supabase.auth.signUp() using anon key
Runs on the Vercel server — GoTrue rate-limits by the Vercel server IP, not the student's campus WiFi IP. Phantom-user detection (empty identities[]) handled server-side. Returns 409 for duplicate emails.
src/app/api/auth/register/route.ts — Step 1
3
Supabase sends confirmation email via Resend SMTP
Custom SMTP (smtp.resend.com:465) bypasses Supabase's 3/hr free-tier email cap. Confirmation link redirects to /auth/callback?token_hash=…&type=signup.
Supabase Auth → SMTP settings → Resend relay
4
Server creates public.users + role profile (same request)
Service-role client inserts into public.users (status: "pending") and either student_profiles or facilitator_profiles. On any DB failure, the auth user is cleaned up via admin.deleteUser() — no orphan accounts.
src/app/api/auth/register/route.ts — Steps 2 & 3
5
User clicks confirmation link → email verified
/auth/callback calls verifyOtp({ token_hash, type: 'signup' }). On success, redirects to /login?verified=true. If link was pre-consumed by an email scanner, shows a safe "already verified" message instead of an error.
src/app/auth/callback/route.ts — Case 1
6
User logs in — 4-step check
① loginUser() → Supabase auth sign-in.
② getAuthUser() → checks email_confirmed_at; unconfirmed users are signed out with a "check your inbox" message.
③ getCurrentUser() → fetches public.users + role profile.
④ Status check: pending → sign out + pending approval screen. rejected → sign out + error. approved → redirect to role dashboard.

API Layer
Method Route Responsibility Auth
POST /api/auth/register Server-side signUp() (anon key, Vercel IP) + public.users insert + role profile. Orphan cleanup on failure. Returns 409 for duplicate emails, 429 for rate limit. None
POST /api/auth/cleanup Deletes a ghost auth user via service-role admin.deleteUser(). Called when profile creation fails after signUp succeeds. None
POST /api/auth/[...all] Handles login (signInWithPassword + httpOnly cookie), logout (clear cookie), and legacy admin register action (admin.createUser + email_confirm: true). None / Session
POST /api/scan Core scan endpoint. Validates QR code ID against student_profiles. Checks student approval status. Enforces time window (present / late / incomplete). Prevents duplicate scans. Records scanned_by. Rate limited: 600/min per IP. Facilitator/Admin
POST /api/events Creates an event, then broadcasts email notifications to all approved students matching target year levels via Resend batch API (100/call). Returns immediately after DB insert — email is fire-and-forget. Rate limited: 20/min. Facilitator/Admin
POST /api/admin/reset-password Admin-triggered password reset. Calls Supabase generateLink({ type: 'recovery' }) and sends the link via Resend email. Link goes to student/facilitator's inbox only — never returned in the response body. Rate limited: 5/min. Admin only
POST /api/admin/settings Reads/writes key-value pairs to system_config table (check-in times, etc.). Admin only
POST /api/admin/test-smtp Sends a test email via Resend to verify SMTP configuration is working. Returns delivery status. Admin only
GET /api/student/profile Returns the authenticated student's full profile including student_profiles join. Student
GET /auth/callback OTP verification (token_hash) and PKCE code exchange. Sets httpOnly cookie on success. Handles email-scanner pre-consumption edge case gracefully. None
Security Model
🔒
RLS on all 7 tables. Students can only read their own records. Facilitators can read/write events and scan records. Admins have broader write access via RLS policies. Service-role client used only in API routes, never exposed to the browser.
🍪
httpOnly cookie session (pharmatrack_token). Access token set on login via API route — never accessible to JavaScript. secure: true, sameSite: lax. 7-day expiry.
🧱
Rate limiting (Upstash Redis sliding window): scan 600/min · register 60/min · reset-password 5/min · events 20/min. Distributed across Vercel instances. Fails open on Redis outage so scan traffic is never blocked by infrastructure.
🌐
signUp() proxied through Vercel server. GoTrue rate-limits by the Vercel IP, not individual student campus WiFi IPs. Removes the shared-IP signup throttle issue for bulk onboarding.
🗑️
No orphan auth users. If DB profile creation fails after signUp(), the server immediately calls admin.deleteUser() via the cleanup route. Students can retry registration cleanly.
📧
Password reset links never returned in API responses. Reset link is sent to the student's inbox only. Admin sees a success toast, never the token. Previous bug (token shown in window.prompt) was fixed in Session 1.
✉️
Email enumeration protection preserved. Phantom user detection (empty identities[]) happens server-side in /api/auth/register, surfaced as a clear "already registered" 409 — consistent with Supabase's own protection, no leakage.
🔍
QR scans validated server-side only. The scan page never writes to Supabase directly. All scans go through POST /api/scan which enforces time windows, checks student approval status, prevents duplicates, and records who scanned.
🧹
Token validation uses getUser() not getSession(). getUser() validates the JWT against the Supabase Auth server on every call — no stale local-storage session can pass. Used in getCurrentUser() throughout all protected pages.
Infrastructure & Services
Supabase
PostgreSQL database, Auth (JWT), Realtime subscriptions. Free tier: 800 MAU / 50,000 cap (1.6% used). DB <10MB / 500MB cap. Custom SMTP configured via Resend. 7 performance indexes applied.
✓ Active · Free tier
Resend
Handles all email delivery: event broadcast notifications (batch API, 100/call), admin password reset links, and Supabase Auth emails (confirmation, recovery). Pro plan: 50,000 emails/mo. Est. 4,000/mo at 5 events.
✓ Active · Pro plan
Upstash Redis
Distributed rate limiting via @upstash/ratelimit sliding window. Shared state across all Vercel function instances. Fail-open fallback to in-memory Map. Set UPSTASH_REDIS_REST_URL + UPSTASH_REDIS_REST_TOKEN in env to activate.
⚠ Env vars needed to activate
Sentry
Error monitoring for client, server, and edge runtimes. Production-only, sendDefaultPii: false, 10% transaction sampling. Source maps uploaded when SENTRY_AUTH_TOKEN is set. Set NEXT_PUBLIC_SENTRY_DSN + SENTRY_DSN to activate.
⚠ DSN env vars needed to activate
Vercel
Next.js 14 App Router deployment. Middleware runs on all function instances. Registration signUp() proxied through server so GoTrue rate-limits by Vercel IP. withSentryConfig wraps the build for source map upload.
✓ Production deployment
QR Code
