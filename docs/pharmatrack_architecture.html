01 — System-Level Architecture
Frontend & Framework
Next.js 14 App Router with server-side rendering and API routes. TypeScript end-to-end. Client components use React 18 with Suspense boundaries and skeleton loaders. Realtime updates via Supabase Postgres Change subscriptions.
Next.js 14React 18TypeScript 5App Router
Database & Auth
Supabase (PostgreSQL) with Row Level Security on all 7 tables. Supabase Auth handles JWT sessions via httpOnly cookie (pharmatrack_token). Auth emails (signup confirmation, password reset) route through Resend SMTP. Service-role client used only in API routes — never exposed to the browser.
Supabase PostgreSQLRLSResend SMTPJWT httpOnly
Rate Limiting
Next.js middleware intercepts four API path prefixes. Upstash Redis sliding window — distributed across all Vercel function instances. Fail-open: if Redis is unreachable, falls back to in-memory Map so scan traffic is never blocked by a Redis outage.
Upstash RedisSliding WindowFail-open
Monitoring & Performance
Sentry integrated (client + server + edge configs). Production-only, sendDefaultPii: false, 10% transaction sampling. 7 PostgreSQL indexes applied on hot columns: attendance_records (student_id, event_id, session_id, event_id+status, created_at), events (date), users (account_type).
Sentry7 DB IndexesRealtime Subscriptions
Email Infrastructure
Resend SDK handles all transactional email. Event broadcasts go out in batches of 100 (8 API calls for 800 students). Fire-and-forget: API returns after DB insert, emails go out async. Auth emails (confirmation, password reset) route through Supabase custom SMTP via Resend — removing the 3 email/hr free-tier cap. Supports 800-student bulk onboarding in one day.
Resend Batch APICustom SMTPFire-and-forget
QR Code
qrcode.react (SVG, Error Correction Level H) for display. html5-qrcode for camera scanning. QR IDs are random 8-char hex (QR-XXXXXXXX) — not derived from user ID, not predictable. PNG export via canvas clone. DOM initialization uses double requestAnimationFrame to prevent race conditions on slow mobile devices.
ECL-HRandom IDshtml5-qrcoderAF DOM init

03 — Authentication Flow — End to End
1
User submits registration form
Browser POSTs to POST /api/auth/register with email, password, full_name, role, and student fields. No browser-side Supabase call.
src/lib/auth-client.ts → registerStudent() / registerFacilitator()
2
Server calls supabase.auth.signUp() using anon key
Runs on the Vercel server — GoTrue rate-limits by the Vercel server IP, not the student's campus WiFi IP. Phantom-user detection (empty identities[]) handled server-side. Returns 409 for duplicate emails.
src/app/api/auth/register/route.ts — Step 1
3
Supabase sends confirmation email via Resend SMTP
Custom SMTP (smtp.resend.com:465) bypasses Supabase's 3/hr free-tier email cap. Confirmation link redirects to /auth/callback?token_hash=…&type=signup.
Supabase Auth → SMTP settings → Resend relay
4
Server creates public.users + role profile atomically
Service-role client inserts into public.users (status: "pending") and either student_profiles or facilitator_profiles. On any DB failure, the auth user is cleaned up via admin.deleteUser() — no orphan accounts.
src/app/api/auth/register/route.ts — Steps 2 & 3
5
User clicks confirmation link → email verified
/auth/callback calls verifyOtp({ token_hash, type: 'signup' }). On success, redirects to /login?verified=true. Email-scanner pre-consumption handled — shows "already verified" message instead of an error.
src/app/auth/callback/route.ts
6
User logs in — 4-step check
① loginUser() → Supabase auth sign-in. ② Checks email_confirmed_at; unconfirmed users are signed out with "check your inbox". ③ getCurrentUser() → fetches public.users + role profile. ④ Status check: pending → sign out + pending screen. rejected → error. approved → redirect to role dashboard.
src/app/login/page.tsx — handleLogin()
7
Admin manually reviews & approves the account
Admin opens User Management, verifies student name, section, and email, clicks Approve. Status changes from pending → approved. One-by-one approval is intentional — prevents incorrectly registered students from slipping through.
src/app/dashboard/admin/users/page.tsx
8
Approved user logs in → role-specific dashboard
Student → /dashboard  ·  Facilitator → /dashboard/facilitator  ·  Admin → /dashboard/admin
Role redirect logic in login/page.tsx

08 — API Layer
Method Route Responsibility Auth
POST /api/auth/register Server-side signUp() (anon key, Vercel IP) + public.users insert + role profile. Orphan cleanup on failure. Returns 409 for duplicate emails, 429 for rate limit. Rate limited: 60/min per IP. None
POST /api/auth/cleanup [Deleted] Was an unauthenticated account-deletion endpoint — anyone could POST a UUID to permanently delete that user. Orphan cleanup is handled inside /api/auth/register. Deleted ✓
POST /api/auth/[...all] Handles login (signInWithPassword + httpOnly cookie), logout (clear cookie), legacy admin register (admin.createUser + email_confirm: true). None / Session
POST /api/scan Core scan endpoint. Validates QR code ID against student_profiles. Checks student approval status. Enforces time window (present/late/incomplete). Prevents duplicate scans. Records scanned_by. Rate limited: 600/min per IP. Facilitator/Admin
POST /api/events Creates an event, broadcasts email notifications to approved students matching target year levels via Resend batch API (100/call). Returns immediately after DB insert — email is fire-and-forget. Rate limited: 20/min. Facilitator/Admin
POST /api/admin/reset-password Generates Supabase recovery link, sends via Resend email. Link goes to the user's inbox only — never in the response body. Rate limited: 5/min. Admin only
POST /api/admin/settings Reads/writes key-value pairs to system_config table (check-in times, etc.). Admin only
POST /api/admin/test-smtp Sends a test email via Resend to verify SMTP configuration. Returns delivery status. Admin only
GET /api/student/profile Returns the authenticated student's full profile including student_profiles join. Student
GET /auth/callback OTP verification (token_hash) and PKCE code exchange. Sets httpOnly cookie. Handles email-scanner pre-consumption edge case. None
09 — Security Model
Critical — Fixed Unauthenticated Account Deletion
/api/auth/cleanup called admin.deleteUser(userId) with zero authentication guard. Anyone on the internet could POST any UUID and permanently delete that auth account.

Fixed: src/app/api/auth/cleanup/route.ts deleted entirely. Orphan cleanup inside /api/auth/register already handled this correctly.
Medium — Fixed userId Override — No Ownership Check
/api/auth/register accepted a userId param without verifying the caller's session matched. An attacker with a victim's UUID could overwrite their profile without their password.

Fixed: When userId is provided without password, getBackendUser(req).id === userId is asserted before proceeding.
src/app/api/auth/register/route.ts:103–108
🔒
RLS on all 7 tables. Students read only their own records. Facilitators read/write events and scan records. Admins have broader write access. Service-role client used only in server-side API routes.
🍪
httpOnly cookie session (pharmatrack_token). Access token set via API route on login — never accessible to JavaScript. secure: true, sameSite: lax. 7-day expiry.
🧱
Upstash Redis sliding window rate limiting (distributed): scan 600/min · register 60/min · auth 20/min · reset-password 5/min · events 20/min. Fails open on Redis outage — scan traffic never blocked by infrastructure.
🌐
signUp() proxied through Vercel server. GoTrue rate-limits by Vercel IP, not individual student campus WiFi IPs. Removes the shared-IP signup throttle for bulk onboarding.
📧
Password reset links never returned in API responses. Reset link is sent to the user's inbox only. Admin sees a success toast, never the token. Prevents token exposure in browser history or JS.
✉️
Email enumeration protection. Phantom user detection (empty identities[]) happens server-side in /api/auth/register, surfaced as a clear 409 "already registered" — never reveals whether the email exists in auth.
🔍
QR scans validated server-side only. Scanner page never writes to Supabase directly. All scans route through POST /api/scan which enforces time windows, checks student approval status, prevents duplicates, and records who scanned.
🧹
Token validation uses getUser() not getSession(). Validates JWT against Supabase Auth server on every call — no stale local-storage session can pass authentication.
🔐
No PII in error monitoring. Sentry configured with sendDefaultPii: false. Production-only. 10% transaction sampling to stay within free tier.

Resend
All email delivery: event broadcast (batch API, 100/call → 8 calls for 800 students), admin password reset links, and Supabase Auth emails (confirmation, recovery). Pro plan: 50,000 emails/mo. Est. ~4,000/mo at 5 events.
✓ Active · Pro plan
Upstash Redis
Distributed rate limiting via @upstash/ratelimit sliding window. Shared state across all Vercel function instances. Fail-open fallback to in-memory Map. Set UPSTASH_REDIS_REST_URL + UPSTASH_REDIS_REST_TOKEN in Vercel env to activate.
⚠ Env vars must be set in Vercel
Sentry
Error monitoring for client, server, and edge runtimes. Production-only, no PII, 10% transaction sampling. Source maps uploaded when SENTRY_AUTH_TOKEN is set. Set NEXT_PUBLIC_SENTRY_DSN + SENTRY_DSN to activate.
⚠ DSN env vars must be set in Vercel
Vercel
Next.js 14 App Router deployment. Middleware runs on all function instances. Registration signUp() proxied through server so GoTrue rate-limits by Vercel IP. withSentryConfig wraps the build for source map upload.
